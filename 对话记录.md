## 根本原因分析

### 数字不会骗人

```
collection: 14.283s  vs  learning: 2.874s   →  5:1 的严重失衡
mean episode length: 42.48 steps  vs  num_steps_per_env: 48
```

**核心问题：episode 几乎全都在 48 步 rollout 内提前终止了**。

### 为什么"CPU 利用率低但 CPU 是瓶颈"

htop 显示 16 核各自只用了 10-25%，但实际上：

- **Python GIL** 使真正的并行执行只占 1-2 个核心（= 总体约 12%）
- 这 1-2 个核心在做**顺序串行的 reset 处理**，卡住了整个 pipeline
- GPU 物理仿真（PhysX）完成后必须等 CPU 侧处理 reset，再继续下一批

**每次 collection cycle 的 reset 数量估算：**
```
40960 envs × (48 / 42.48 steps) ≈ 46,300 次 reset/cycle
```
每次 reset 需要在 CPU 侧执行：运动数据采样 → 初始化机器人姿态 → 写回 GPU tensor。  
这就是为什么 CPU 看起来"闲"，但其实串行地处理几万次 reset，让 GPU 在等它。

### 导致频繁 reset 的罪魁祸首


## `num_steps_per_env` 调大的连锁影响

### 影响1：minibatch size 翻倍（最关键）

```
buffer_size = num_envs × num_steps_per_env
minibatch_size = buffer_size / num_mini_batches

当前：  40960 × 48  / 40 = 49,152  per minibatch
调到96: 40960 × 96  / 40 = 98,304  per minibatch  ← 翻倍
```

minibatch 过大会导致：梯度更新次数减少（相同 epoch 内滑过更少批次）。  
**解决**：`num_mini_batches` 同比例增加：

```python
# 48 → 96 时
num_steps_per_env = 96
num_mini_batches  = 80   # 40 → 80，保持 minibatch size 不变
```

---

### 影响2：GPU VRAM 占用线性增长

buffer 存的是 `(obs, action, reward, done, value, log_prob)` 的完整 transition，全量放在 GPU。  

|num_steps|总 transitions|相对当前|
|---|---|---|
|48 (当前)|~1.97M|1x|
|96|~3.93M|2x|
|128|~5.24M|2.7x|

如果 VRAM 不够会 OOM。可以同时适当缩小 `num_envs` 来补偿：

```bash
# 保持 buffer 总量近似不变
--num_steps_per_env 96 --num_envs 20480  # buffer: 20480×96 ≈ 1.97M
```

---

### 影响3：gamma / lam **不需要改**

框架里 `LowFreqPPORunnerCfg` 调整 gamma/lam 是因为它同时修改了 `decimation`（控制频率变了，折扣步长的物理时间变了）。  
你只改 rollout 长度，**控制步长不变**（50Hz 不变），gamma/lam 含义相同，无需动。

---

### 影响4：advantage 估计变好（正面）

rollout 48 步只覆盖约 0.96s，不够看清 delayed reward 的效果。  
96 步 = 1.92s ≈ 接近 `1/(1-γ)=100` 步的有效折扣窗口，GAE 质量会明显提升，有助于训练稳定性。
